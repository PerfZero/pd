ТЕХНИЧЕСКОЕ ЗАДАНИЕ
1. ОБЩИЕ ПОЛОЖЕНИЯ
1.1. Цель и предмет работ
Исполнитель обязуется выполнить доработку корпоративного веб-портала PassDesk в соответствии с требованиями настоящего технического задания.

Доработка включает:
·         оптимизацию существующего кодовой базы и архитектуры приложения
·         внедрение систем автоматизированного распознавания документов (OCR)
·         интеграцию с внешними системами: Реестр контролируемых лиц МВД РФ, система контроля и управления доступом (СКУД)
·         разработку модуля "Охрана труда" с системой управления документами и статусами подрядчиков
·         разработку мобильных приложений для iOS и Android
·         разработку Telegram-бота для генерации QR-кодов
·         расширение функционала управления пользователями, сотрудниками и документами
·         внедрение аналитического модуля для отчетности и мониторинга
·         добавление многоязычного интерфейса

1.2. Объект автоматизации
Корпоративный веб-портал PassDesk для управления пропусками и кадровым учетом сотрудников генподрядной организации и субподрядных организаций.

Текущий технологический стек:
·         Frontend: React 18, Vite, Ant Design, адаптивный дизайн (desktop, tablet, mobile)
·         Backend: Node.js v18+, Express.js
·         База данных: PostgreSQL (Yandex Managed Service for PostgreSQL)
·         Хранилище файлов: Yandex Object Storage (S3-совместимое)
·         Архитектура: Feature-Sliced Design (FSD)
·         Развертывание: PM2, Nginx, SSL (Let's Encrypt)


1.3. Термины и определения
СКУД — Система контроля и управления доступом (торговая марка "СИГУР" или аналог)
ОТ — Охрана труда
OCR — Optical Character Recognition, оптическое распознавание символов
РКЛ — Реестр контролируемых лиц МВД РФ
ИТР — Инженерно-технические работники
QR-код — динамически генерируемый код для авторизации прохода сотрудника через СКУД
Signed URL — подписанная ссылка для доступа к файлам в S3-хранилище с ограниченным сроком действия
Soft Delete — мягкое удаление записей с установкой флага is_deleted без физического удаления из БД

2. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
2.1. Аудит, рефакторинг и оптимизация кодовой базы

Цель: Повышение читаемости, поддерживаемости и производительности кода.

Требования:

·         Удалить неиспользуемые функции, импорты и закомментированные участки кода.
·         Оптимизировать рендеринг компонентов, уменьшить количество лишних перерисовок.

2.1.2. Аудит безопасности
Требования:

Защита API:
·         Проверить все API-методы на наличие middleware проверки прав доступа (authorize, проверка ролей)
·         Добавить логирование всех попыток несанкционированного доступа к информации/файлам в таблицу

2.1.3. Оптимизация базы данных
Требования:

·         Оптимизировать запросы с JOIN: исключить N+1 проблемы, использовать WHERE-условия перед JOIN

2.2. Распознавание документов (OCR) и автоматическое заполнение форм
2.2.1. Интеграция с сервисом OCR
Цель: Автоматизация ввода данных документов сотрудников.

Требования:
Интегрировать API сервис OCR для распознавания следующих типов документов:
·         Паспорт гражданина РФ (серия, номер, дата выдачи, кем выдан, дата рождения, ФИО)
·         Паспорт иностранного гражданина
·         Патент на работу
·         Виза
·         КИГ (Карта иностранного гражданина)
·         API-ключи хранить в переменных окружения .env на сервере (OCR_API_KEY)
·         Обеспечить безопасную передачу файлов в сервис OCR (использовать HTTPS)

2.2.2. Сценарий работы
Шаг 1. Загрузка документа:
·         Пользователь загружает скан или фото документа в карточку сотрудника (раздел "Документы")
·         Кнопка "Распознать документ" становится активной после успешной загрузки файла

Шаг 2. Отправка на распознавание:
·         При нажатии кнопки файл отправляется на API OCR
·         Во время обработки отображается индикатор загрузки (spinner)
·         Результат распознавания возвращается в формате JSON

Шаг 3. Автозаполнение полей формы:
·         Если поля формы пустые — автоматически заполняются данными из OCR
·         Если поля уже заполнены, но данные отличаются от результата OCR:
·             Поля подсвечиваются оранжевым цветом
·             Рядом с полем отображается значок "⚠️" с всплывающей подсказкой: "Данные отличаются от распознанных. Проверьте правильность"
·             Пользователь может выбрать: оставить текущее значение или заменить на распознанное (кнопки "Оставить" / "Заменить")

Шаг 4. Сохранение:
·         После подтверждения данные сохраняются в БД
·         В таблице добавляется поле ocr_verified (boolean) для отметки, что документ был проверен через OCR

2.2.3. Доработка мобильного интерфейса для OCR
Требования:
·         В мобильной версии портала улучшить работу функции "Фотографирование в режиме документа":
·             Автоматическое обнаружение краев документа (оптимизировать анализ, расширить рамку)
·             Автоматическая коррекция перспективы
·             Возможность повторной съемки перед отправкой
·         После съемки файл автоматически отправляется на сервер для OCR-распознавания
·         Результат распознавания отображается на экране редактирования сотрудника

2.3. Контроль сроков действия документов
2.3.1. Автоматическая проверка истечения сроков - доработать
Требования:
·         Для документов с ограниченным сроком действия (паспорт РФ, патент, виза, КИГ) доработать автоматический контроль:
·             Ежесуточная проверка через Cron Job (время выполнения: 06:00 МСК)
·             При истечении срока действия документа:
·                 Статус сотрудника автоматически изменяется на "Заблокирован"
·                 Доступ в СКУД блокируется (отправляется команда в СКУД)
·                 Генерируется уведомление для менеджера контрагента-владельца и контрагента-сотрудника

2.3.2. Уведомления за 2 недели до истечения - доработать
Требования:
·         За 14 дней до истечения срока документа отправлять уведомление:
·             Менеджеру контрагента-владельца сотрудника (email + уведомление в портале)
·             Менеджеру контрагента-сотрудника (email + уведомление в портале)
·         Формат уведомления:
·             Тема: "Истекает срок действия документа сотрудника [ФИО]"
·             Содержание: Тип документа, дата истечения, ФИО сотрудника, контрагент
·         В личном кабинете пользователя отображать список сотрудников с истекающими документами

2.4. Проверка в Реестре контролируемых лиц (РКЛ) МВД РФ
2.4.1. Интеграция с API МВД
Требования:
·         Интегрировать сервис проверки по Реестру контролируемых лиц МВД РФ через официальное API (или через сертифицированный сервис-посредник)
·         API-ключи и параметры доступа хранить в .env файле (MVD_API_URL, MVD_API_KEY)

2.4.2. Алгоритм проверки
Входные данные для запроса:
·         ФИО (полностью)
·         Дата рождения
·         Серия и номер паспорта
·         Гражданство

Триггеры запуска проверки:
·         1. Автоматически при создании или редактировании карточки сотрудника (если изменились паспортные данные)
·         2. По расписанию (Cron Job): ежесуточная проверка всех активных сотрудников (время выполнения: 02:00 МСК)
·         3. Вручную: кнопка "Проверить в РКЛ" в карточке сотрудника (доступна ролям: Admin, Manager, ИнженерОТ)

Обработка результата:
·         Если сотрудник найден в РКЛ:
·             Статус сотрудника изменяется на "Заблокирован (РКЛ)"
·             Доступ в СКУД блокируется
·             Уведомление отправляется администратору портала и менеджеру контрагента
·         Если сотрудник не найден:
·             Добавляется отметка rkl_checked_at (timestamp) в таблицу employees
·             Статус остается без изменений
·         Все запросы логируются в таблицу rkl_check_logs (дата, время, результат, ФИО, документ)

2.4.3. Доработка системы статусов
Требования:
·         Провести аудит существующих статусов сотрудников:
·             Удалить неиспользуемые статусы
·             Объединить дублирующиеся статусы
·         Реализовать управление статусами через веб-интерфейс (раздел "Администрирование" → "Статусы сотрудников"):
·             Добавление нового статуса (название, описание, цветовая метка)
·             Редактирование существующего статуса
·             Деактивация статуса (soft delete)
·         Привязка статусов:
·             Глобальные статусы — применяются к сотруднику независимо от контрагента (например: "Заблокирован (РКЛ)", "Заблокирован (Истек документ)")
·             Контекстные статусы — привязаны к связке "Сотрудник + Контрагент" ("Уволен", "Деактивирован")
·         При изменении статуса создавать новую запись в таблице employee_status_history (сотрудник, статус, дата изменения, инициатор), старая запись переводится в is_active = false
·         Добавить статус для выгруженных согласий на обработку персональных данных

2.5. Модуль "Охрана труда" (ОТ)
2.5.1. Общие положения
Цель: Организация системы управления документами по охране труда для контрагентов-подрядчиков.

Требования:
·         Создать отдельный раздел "Охрана труда" в боковом меню портала (страница)
·         Доступ к разделу:
·             Администратор ОТ — полный доступ (управление категориями, документами, бланками, проверка документов)
·             Инженер ОТ — просмотр и проверка документов, загрузка от имени подрядчика
·             Администратор портала — те же права, что у Администратора ОТ
·             User (контрагент-подрядчик) — доступ только к вкладке "Подрядчик" (загрузка документов своей организации)
·             Менеджер и User контрагента "default" — доступа не имеют

2.5.2. Структура раздела
Раздел содержит 4 вкладки:
·         1. Все объекты — сводная таблица по всем объектам
·         2. Объект — детальная информация по выбранному объекту
·         3. Подрядчик — иерархическая структура категорий и документов
·         4. Настройка — управление категориями, документами, бланками (доступна только Администратору ОТ и Администратору портала)

2.5.3. Вкладка "Подрядчик"
Структура отображения:
1.       1. Категория 1     1.1. Подкатегория 1.1        1.1.1. Подподкатегория 1.1.1       	1.1.1.1. Документ 1 — [Бланк] [Загрузить]       	1.1.1.2. Документ 2 — [Бланк] [Загрузить]     1.2. Подкатегория 1.2    	...  2. Категория 2     ...

Функционал по ролям:

Администратор ОТ / Администратор портала:
·         Добавление, редактирование, удаление категорий и документов
·         Загрузка бланков (шаблонов) документов
·         Назначение обязательности документов для категории
·         Скачивание и просмотр загруженных документов
·         Подтверждение/отклонение документов, загруженных подрядчиками
·         Загрузка документов от имени подрядчика

Инженер ОТ:
·         Скачивание и просмотр загруженных документов и бланков
·         Подтверждение/отклонение документов
·         Загрузка документов от имени подрядчика

User (подрядчик):
·         Скачивание бланков документов
·         Загрузка сканов/фото документов
·         Просмотр статуса проверки документов (загружен / проверяется / подтвержден / отклонен)
·         Удаление собственных документов невозможно (только замена новой версией)

Отображение документов:

·         Название документа
·         Иконка "Бланк" (если загружен шаблон) — при клике скачивается файл
·         Кнопка "Загрузить" (для подрядчика)
·         Статус документа (цветовая индикация):
·             🔴 Красный — не загружен
·             🟠 Оранжевый — загружен, не проверен
·             🟢 Зеленый — загружен и подтвержден
·             ⚫ Серый — отклонен (с комментарием причины)
·         Для инженеров/администраторов — кнопки "✅ Подтвердить" / "❌ Отклонить"

Фильтры:
·         По объектам (выпадающий список)
·         По подрядчикам, связанным с выбранным объектом

2.5.4. Вкладка "Объект"
Функционал:
·         При входе на вкладку — выпадающий список для выбора объекта строительства
·         После выбора объекта отображается таблица с подрядчиками и статусом заполнения документов по категориям
·         Категории с обязательными документами: отображаются статусом "Заполнен" (зеленый) / "Не заполнен" (красный)
·         Категории без обязательных документов: отображается дробь "загружено/всего"
·         Допуск:
·             Допущен — все обязательные документы загружены и подтверждены
·             Не допущен — есть незагруженные или отклоненные обязательные документы
·             Временно допущен — выставляется вручную инженером/администратором ОТ
·         При клике на строку подрядчика — переход на вкладку "Подрядчик" с примененными фильтрами (Объект + Подрядчик)

2.5.5. Вкладка "Все объекты"
Функционал:
·         Сводная таблица по всем объектам строительства с количеством допущенных, не допущенных и временно допущенных подрядчиков

Взаимодействие:
·         При клике на название объекта — переход на вкладку "Объект" с выбранным объектом

2.5.6. Вкладка "Настройка"
Доступ: только Администратор ОТ и Администратор портала.

Функционал:

Управление категориями:
·         Создание новой категории (название, описание, родительская категория)
·         Редактирование категории
·         Удаление категории (soft delete)
·         Изменение порядка отображения (drag-and-drop)

Управление документами:
·         Добавление нового типа документа в категорию
·         Установка признака "Обязательный документ" (checkbox)
·         Загрузка бланка (шаблона) документа
·         Редактирование, удаление (soft delete)

Управление библиотекой шаблонов:
·         Создание библиотеки готовых шаблонов документов (Приказы, Доверенности, Протоколы и т.д.)
·         Возможность скачивания шаблонов всеми пользователями раздела ОТ

Инструкции для подрядчиков:
·         Текстовый блок с инструкциями по заполнению документов
·         Возможность загрузки файла-инструкции (PDF, DOCX)

2.5.7. Статусы контрагентов в модуле ОТ
Требования:
·         Статусы рассчитываются автоматически по формуле:
·             Допущен: Все обязательные документы в статусе "Подтвержден"
·             Не допущен: Есть хотя бы один обязательный документ в статусе "Не загружен" или "Отклонен"
·             Временно допущен: Выставляется вручную инженером/администратором (переопределяет автоматический расчет)
·         Статусы привязаны к связке Контрагент + Объект
·         При изменении статуса создается новая запись в таблице ot_contractor_status_history (контрагент, объект, статус, дата изменения, инициатор), старая запись переводится в is_active = false

Уведомления:
·         При изменении статуса на "Не допущен" отправлять уведомление менеджеру контрагента:
·             Email
·             Уведомление в портале (всплывающее при входе)
·             Содержание: Объект, причина (список недостающих документов)

2.5.8. Комментарии по документам и подрядчикам
Требования:
·         Реализовать систему комментариев:
·             Комментарии к документу: инженер/администратор может оставить комментарий при отклонении документа (причина отклонения)
·             Комментарии к подрядчику: общий комментарий по работе подрядчика на объекте (доступен только инженерам и администраторам)
·         Комментарии отображаются в виде хронологического списка с указанием автора и даты

2.6. Интеграция с СКУД "СИГУР" (или аналогичной системой)
2.6.1. Архитектура интеграции
Требования:
·         Разработать модуль-коннектор на Node.js (или аналог) для взаимодействия с сервером СКУД
·         Использовать протокол интеграции СКУД (Integration Protocol / Styx REST API или API, предоставляемый службой поддержки СИГУР)
·         Коннектор должен работать как отдельный сервис (микросервис) или модуль внутри основного сервера
·         Поддержка очереди запросов (использовать Bull + Redis) для обработки большого потока событий проходов

Схема взаимодействия:

1. Управление доступом (PassDesk → СКУД):
·         При одобрении пропуска в PassDesk:
·             Создается/обновляется профиль сотрудника в СКУД (ФИО, контрагент)
·             Активируется доступ на указанный период (дата начала, дата окончания)
·         При блокировке/увольнении:
·             Отправляется команда на блокировку доступа в СКУД
·         При попадании в "Черный список" (РКЛ):
·             Немедленная блокировка доступа во всех СКУД

2. Авторизация по QR-коду (СКУД → PassDesk):
·         Сотрудник сканирует QR-код на турникете
·         СКУД отправляет запрос на выделенный порт портала PassDesk для проверки валидности QR-кода
·         Портал проверяет QR-код (валидность, срок действия, статус сотрудника) и отвечает синхронно с разрешением или отказом в доступе

3. Регистрация событий проходов (СКУД → PassDesk):
·         СКУД отправляет события входа/выхода на эндпоинт портала
·         Портал сохраняет событие в таблицу (например access_events)

2.6.2. Генерация QR-кодов
Требования:
·         QR-код генерируется динамически для каждого запроса
·         QR-код содержит:
·             Уникальный идентификатор сотрудника
·             Timestamp генерации
·             Подпись (HMAC) для защиты от подделки
·         Срок действия QR-кода:
·             Для постоянных сотрудников: 1 час (обновляется по запросу)
·             Для временных пропусков (генерируется кадровой службой): одноразовый
·         QR-код генерируется:
·             Через мобильное приложение (iOS/Android)
·             Через Telegram-бот
·             Через веб-интерфейс портала (для кадровой службы)

2.6.3. Управление физическими картами доступа
Требования:
При выдаче физической карты доступа:
·             Номер карты хранится на сервере СКУД.
·             Данные передаются в СКУД для разрешения прохода.

3. ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ
3.1. Backend
·         Платформа: Node.js v18+
·         Фреймворк: Express.js
·         База данных: PostgreSQL (используется существующий Managed Service от Yandex Cloud)
·         ORM: Sequelize (для миграций и работы с БД)
·         Очереди задач: Bull + Redis (для долгих операций: OCR, запросы к МВД, синхронизация СКУД)
·         WebSockets: Socket.io (для real-time обновлений)
·         Логирование: Winston или Pino (расширенное логирование с ротацией файлов)
·         Аутентификация: JWT (JSON Web Tokens), refresh tokens
·         Валидация: Joi или express-validator
·         Безопасность: Helmet, CORS, Rate Limiting (express-rate-limit)

3.2. Frontend
·         Платформа: React 18
·         Сборщик: Vite
·         UI-библиотека: Ant Design (используется в текущей версии)
·         Графики: @ant-design/charts или Recharts
·         Управление состоянием: Redux Toolkit или Zustand
·         HTTP-клиент: Axios
·         Локализация: react-i18next
·         Real-time: Socket.io-client или SWR/React Query
·         Адаптивность: CSS Grid, Flexbox, media queries

3.3. Логирование
Требования:
·         Логирование всех действий пользователей:
·             Вход/выход из системы
·             Создание/редактирование/удаление сотрудников
·             Загрузка/удаление документов
·             Изменение статусов
·             Запросы к внешним API (OCR, МВД, СКУД)
·         Уровни логирования: error, warn, info, debug
·         Логи хранятся в файлах (ротация ежедневно, хранение 30 дней) и в таблице БД system_logs (для критических событий)
·         Формат логов: JSON

3.4. Мониторинг и производительность
Требования:
·         Настроить PM2 для управления Node.js процессами (уже используется)
·         Мониторинг:
·             PM2 Plus (dashboard для мониторинга процессов)
·             PostgreSQL: установить расширение pg_stat_statements для мониторинга запросов
·             Redis: использовать Redis Commander для мониторинга очередей
·         Метрики:
·             Время ответа API (response time)
·             Количество запросов в секунду (RPS)
