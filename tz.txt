2.2. Распознавание документов (OCR) и автоматическое заполнение форм
2.2.1. Интеграция с сервисом OCR
Цель: Автоматизация ввода данных документов сотрудников.

Требования:
Интегрировать API сервис OCR для распознавания следующих типов документов:
·         Паспорт гражданина РФ (серия, номер, дата выдачи, кем выдан, дата рождения, ФИО)
·         Паспорт иностранного гражданина

·         Виза

·         API-ключи хранить в переменных окружения .env на сервере (OCR_API_KEY)
·         Обеспечить безопасную передачу файлов в сервис OCR (использовать HTTPS)

2.2.2. Сценарий работы
Шаг 1. Загрузка документа:
·         Пользователь загружает скан или фото документа в карточку сотрудника (раздел "Документы")
·         Кнопка "Распознать документ" становится активной после успешной загрузки файла

Шаг 2. Отправка на распознавание:
·         При нажатии кнопки файл отправляется на API OCR
·         Во время обработки отображается индикатор загрузки (spinner)
·         Результат распознавания возвращается в формате JSON

Шаг 3. Автозаполнение полей формы:
·         Если поля формы пустые — автоматически заполняются данными из OCR
·         Если поля уже заполнены, но данные отличаются от результата OCR:
·             Поля подсвечиваются оранжевым цветом
·             Рядом с полем отображается значок "⚠️" с всплывающей подсказкой: "Данные отличаются от распознанных. Проверьте правильность"
·             Пользователь может выбрать: оставить текущее значение или заменить на распознанное (кнопки "Оставить" / "Заменить")

Шаг 4. Сохранение:
·         После подтверждения данные сохраняются в БД
·         В таблице добавляется поле ocr_verified (boolean) для отметки, что документ был проверен через OCR

·         API-ключи и параметры доступа хранить в .env файле (MVD_API_URL, MVD_API_KEY)

2.4.2. Алгоритм проверки
Входные данные для запроса:
·         ФИО (полностью)
·         Дата рождения
·         Серия и номер паспорта
·         Гражданство

Триггеры запуска проверки:
·         1. Автоматически при создании или редактировании карточки сотрудника (если изменились паспортные данные)
·         2. По расписанию (Cron Job): ежесуточная проверка всех активных сотрудников (время выполнения: 02:00 МСК)
·         3. Вручную: кнопка "Проверить в РКЛ" в карточке сотрудника (доступна ролям: Admin, Manager, ИнженерОТ)

Обработка результата:
·         Если сотрудник найден в РКЛ:
·             Статус сотрудника изменяется на "Заблокирован (РКЛ)"
·             Доступ в СКУД блокируется
·             Уведомление отправляется администратору портала и менеджеру контрагента
·         Если сотрудник не найден:
·             Добавляется отметка rkl_checked_at (timestamp) в таблицу employees
·             Статус остается без изменений
·         Все запросы логируются в таблицу rkl_check_logs (дата, время, результат, ФИО, документ)

Требования:
·         Провести аудит существующих статусов сотрудников:
·             Удалить неиспользуемые статусы
·             Объединить дублирующиеся статусы
·         Реализовать управление статусами через веб-интерфейс (раздел "Администрирование" → "Статусы сотрудников"):
·             Добавление нового статуса (название, описание, цветовая метка)
·             Редактирование существующего статуса
·             Деактивация статуса (soft delete)
·         Привязка статусов:
·             Глобальные статусы — применяются к сотруднику независимо от контрагента (например: "Заблокирован (РКЛ)", "Заблокирован (Истек документ)")
·             Контекстные статусы — привязаны к связке "Сотрудник + Контрагент" ("Уволен", "Деактивирован")
·         При изменении статуса создавать новую запись в таблице employee_status_history (сотрудник, статус, дата изменения, инициатор), старая запись переводится в is_active = false
·         Добавить статус для выгруженных согласий на обработку персональных данных

2.6. Интеграция с СКУД "СИГУР" (или аналогичной системой)
2.6.1. Архитектура интеграции
Требования:
·         Разработать модуль-коннектор на Node.js (или аналог) для взаимодействия с сервером СКУД
·         Использовать протокол интеграции СКУД (Integration Protocol / Styx REST API или API, предоставляемый службой поддержки СИГУР)
·         Коннектор должен работать как отдельный сервис (микросервис) или модуль внутри основного сервера
·         Поддержка очереди запросов (использовать Bull + Redis) для обработки большого потока событий проходов

Схема взаимодействия:

1. Управление доступом (PassDesk → СКУД):
·         При одобрении пропуска в PassDesk:
·             Создается/обновляется профиль сотрудника в СКУД (ФИО, контрагент)
·             Активируется доступ на указанный период (дата начала, дата окончания)
·         При блокировке/увольнении:
·             Отправляется команда на блокировку доступа в СКУД
·         При попадании в "Черный список" (РКЛ):
·             Немедленная блокировка доступа во всех СКУД

2. Авторизация по QR-коду (СКУД → PassDesk):
·         Сотрудник сканирует QR-код на турникете
·         СКУД отправляет запрос на выделенный порт портала PassDesk для проверки валидности QR-кода
·         Портал проверяет QR-код (валидность, срок действия, статус сотрудника) и отвечает синхронно с разрешением или отказом в доступе

3. Регистрация событий проходов (СКУД → PassDesk):
·         СКУД отправляет события входа/выхода на эндпоинт портала
·         Портал сохраняет событие в таблицу (например access_events)

2.6.2. Генерация QR-кодов
Требования:
·         QR-код генерируется динамически для каждого запроса
·         QR-код содержит:
·             Уникальный идентификатор сотрудника
·             Timestamp генерации
·             Подпись (HMAC) для защиты от подделки
·         Срок действия QR-кода:
·             Для постоянных сотрудников: 1 час (обновляется по запросу)
·             Для временных пропусков (генерируется кадровой службой): одноразовый
·         QR-код генерируется:
·             Через мобильное приложение (iOS/Android)
·             Через Telegram-бот
·             Через веб-интерфейс портала (для кадровой службы)

2.6.3. Управление физическими картами доступа
Требования:
При выдаче физической карты доступа:
·             Номер карты хранится на сервере СКУД.
·             Данные передаются в СКУД для разрешения прохода.

2.15.1. Привязка статусов к контрагенту
Требования:
·         Реализовать поддержку работы сотрудника в нескольких организациях (контрагентах)
·         Статусы "Уволен", "Деактивирован" привязываются к связке "Сотрудник + Контрагент"
·         Сотрудник может быть уволен у одного контрагента, но работать у другого
·         В карточке сотрудника отображать список контрагентов с текущими статусами

2.16. Real-time обновления интерфейса
2.16.1. Требования
Цель: Обеспечить автоматическое обновление данных на странице без перезагрузки.

События для real-time обновления:
·         Добавление нового сотрудника
·         Изменение статуса сотрудника
·         Загрузка документа
·         Подтверждение/отклонение документа (модуль ОТ)
·         Событие прохода через СКУД (для аналитики в реальном времени)

2.17. Таблица документов
2.17.1. Требования
Цель: Централизованный просмотр всех документов сотрудников контрагента.

Функционал:
·         Новая страница "Документы контрагента"
·         Фильтры:
·             Контрагент (выпадающий список)
·             Тип документа (Паспорт, Патент, Виза и т.д.)
·             Статус документа (Загружен, Не загружен, Проверен OCR, Срок истекает)
·             Сотрудник (поиск по ФИО)
·         Таблица:
·             ФИО сотрудника, Тип документа, Серия/Номер, Дата выдачи, Срок действия, Статус, Действия (Скачать, Удалить)
·         Массовая выгрузка:
·             Кнопка "Скачать все документы" (архив ZIP)
·             Кнопка "Экспортировать в Excel" (таблица с метаданными документов)
·         Ввести обязательность документов (так же как обязательность полей – редактируемую)
